# 我和 Claude做了一个 VSCode Markdown 编辑器

> **副标题：一次真实的 AI 协作开发体验**
>
> 作者：liuyaoming · 2026-03-25

## 一、为什么做这个

每次写 Markdown，都要在"源码编辑模式"和"预览模式"来回切换——写了一段 Markdown 语法，得切过去看效果；觉得排版不对，再切回来改。市面上也有类似的 WYSIWYG 插件，但用起来总有些不顺手。

于是想：**能不能用 AI 辅助，从 0 开始做一个自己满意的？**

> **该项目 99% 的代码由 Claude 生成。**
> 人的角色是：提需求、看效果、发现问题、把控方向。

***

## 二、项目概览

**一句话**：接管 VSCode 对 `.md` 文件的打开，直接在 WYSIWYG 界面编辑，支持 GFM 全语法。

### 技术栈

| 层级          | 选型                          | 说明                            |
| ----------- | --------------------------- | ----------------------------- |
| VSCode API  | `CustomEditorProvider`      | 接管 .md 文件打开，支持读写              |
| WYSIWYG 编辑器 | Milkdown v7（基于 ProseMirror） | 原生 Markdown 序列化，GFM 支持        |
| 构建工具        | esbuild                     | 双目标构建（Node.js + Browser），< 1s |
| 语言          | TypeScript                  | Extension + WebView 全部 TS     |

### 架构

```
VSCode Extension 进程（Node.js）
    ↕ 消息协议（messaging.ts）
WebView 前端（Browser）
    └── Milkdown 编辑器
        ├── tableHandles.ts   （拖拽 handle）
        ├── tableAddButtons.ts（插入线）
        ├── selectionToolbar.ts（浮动工具栏）
        ├── codeBlockView.ts  （代码块 UI）
        ├── toc.ts            （目录面板）
        └── linkPopup.ts      （链接弹窗）
```

## 四、与 Claude 协作的真实体验

这是本次分享最想聊的部分。

### 4.1 效率最高的场景

**需求清晰 → Claude 直接出完整实现。**

比如表格插入线：我描述"鼠标悬浮在行/列边框时出现绿色高亮线和 + 号，点击插入行列"，Claude 给出了完整的 `mousemove` 坐标检测 + `overlay` 元素定位的实现，一次可用。

类似的还有：

* 粘贴报错堆栈 → Claude 精准定位根因

* "把这个颜色换成绿色"/"阈值从 4 改到 8" 这类微调，几乎没有沟通成本

### 4.2 需要人来把控的场景

* **需求描述模糊时**：Claude 会"猜"一个方向，不一定对。比如"表格选中效果不好看"，Claude 可能猜成改颜色，但真正的问题可能是触发区域逻辑错了

* **视觉效果判断**：表格选中样式（绿色边框 vs 纯背景 vs 两者都有）改了 5\~6 次——这种"好不好看"只有人能判断

* **技术选型**：要不要引外部库、要不要动底层框架，Claude 会给选项，但决策必须人来

### 4.3 最有意思的一段：B054\~B056，改了 4 次的 bug

多格拖拽选中松手后，选中效果有小概率消失。这个 bug 前后修了 4 次，每次 Claude 都觉得"这次彻底了"：

```
第 1 次：新增 wasCrossCell 标志，同步清除 pendingClickPos
   → 还有小概率复现

第 2 次：格内拖拽分支加同格检查（anchor/head 必须在同一格）
   → 还有极小概率复现

第 3 次：微任务恢复机制（lastGoodCellSelection 保存）
   → 仍有复现

第 4 次：filterTransaction 拦截，200ms 保护窗口
   → 彻底解决 ✅
```

<br />

### 4.4 典型案例一：插入线判定区域调试（可视化让问题一目了然）

表格插入线的触发判定始终不理想——靠近单元格边框悬浮，插入线经常不出现。Claude 一轮轮调整判定阈值，效果没有本质改善，最后甚至把整个表格都设成了触发区域。

**我的介入**：提出加个调试模式——**用红色边框把判定区域画出来，我来肉眼看**。

加上可视化后，立刻发现根因：

> 除第一列外，其他单元格的判定区域**只在右侧和下边**，左侧和上边完全没有判定。
> 所以靠近左边缘或上边缘时，插入线永远不出现。

**教训：遇到"调了很多次还是不对"的 bug，先把问题可视化，比继续调参有效 10 倍。**

### 4.5 典型案例二：多格选中小概率失效（日志定位法）

多格选中偶发失效，复现概率很低，Claude 改了几轮都没找到真正的触发路径——因为只靠"现象描述"，信息太少。

**我的介入**：让 Claude 加调试日志，记录：

* 选中开始坐标

* 选中结束坐标

* 选中了几个格

* **取消选中的来源（含完整调用堆栈）**

测试几次后，把"正常选中"和"选中失效"两份日志**同时给 Claude 对比**。

Claude 看到堆栈后，**一次就找到了根因**：ProseMirror 内部的 `readDOMChange` 通过 `setTimeout` 延迟执行，在选中后用浏览器原生 DOM 选区覆盖了 CellSelection。

**教训：小概率复现的 bug，加详细日志（尤其调用栈）喂给 AI，比口头描述现象效率高很多。**

## 五、技术难点：ProseMirror 的异步陷阱

> 这节偏技术，可以快速过一下背景就跳过。

### 5.1 为什么表格单击会出现绿色闪烁

```
Milkdown GFM preset 配置了 allowTableNodeSelection: true
 → 单击表格单元格 → NodeSelection(cell)
 → prosemirror-tables normalizeSelection 自动转换
 → CellSelection（绿色高亮背景）
 → 下一个事件才纠正 → 用户看到绿色闪一下
```

**解决**：在 `appendTransaction`（每次 state 更新后、DOM 渲染前同步执行）里检测单格 CellSelection，立即改为 TextSelection。用户感知不到任何闪烁。

### 5.2 更难的：为什么多格拖拽选区会偶发消失

```
ProseMirror 原生 mouseDown.up()
 → 在 mouseup 冒泡阶段读取浏览器 DOM 原生选区
 → 浏览器不理解 CellSelection
 → createSelectionBetween 偶发失败
 → dispatch TextSelection 覆盖正确的 CellSelection
```

关键在于 `readDOMChange` 走的是 `setTimeout`——它在所有微任务都跑完之后才执行，所以任何微任务级别的恢复机制都拦不住它。

**最终解决**：`filterTransaction` 在事务层拦截所有 dispatch，200ms 保护窗口内拒绝来自 `readDOMChange` 的 TextSelection。

## 六、Bug 统计与反思

### 6.1 总体数据

| 指标       | 数量                     |
| -------- | ---------------------- |
| 开发会话     | 25 次                   |
| 总 Bug    | **63 个**（B001 \~ B063） |
| 已修复      | 60 个 ✅                 |
| 反复出现（🔄） | 5 个                    |
| 待处理（⏳）   | 3 个                    |

### 6.2 反复修改最多的 Bug TOP 5

| Bug       | 描述                    | 出现次数       | 最终方案                               |
| --------- | --------------------- | ---------- | ---------------------------------- |
| B062      | CellSelection 行号只返回单行 | **8 次**    | `$anchorCell.pos` + `Math.min/max` |
| B056      | 多格拖拽松手后选区消失           | 5 次，修了 4 次 | `filterTransaction` 200ms 拦截       |
| B054/B055 | 多格拖拽系列前驱 bug          | 各 4\~5 次   | `wasCrossCell` + 同格检查              |
| B023      | 行号映射精度                | 4 次        | textSearch + verbatim 降级           |
| B016      | 发送到 Claude 无效果        | 3 次        | 修复 `insertAtMention` 参数格式          |

### 6.3 Bug 分布（大致）

| 类型                         | 占比    | 典型例子                                  |
| -------------------------- | ----- | ------------------------------------- |
| 视觉 / CSS（颜色/位置/闪烁）         | \~25% | 表格选中样式迭代、插入线位置                        |
| 功能逻辑（行号/发送/保存）             | \~30% | 行号映射、sendToClaudeChat                 |
| 交互 / 事件竞争                  | \~20% | CellSelection 系列                      |
| 框架适配（ProseMirror/Milkdown） | \~25% | allowTableNodeSelection、readDOMChange |

### 6.4 反思

1. **视觉方案要早确定**：表格选中样式改了 5\~6 次，每次改动都带动 CSS 和交互逻辑联动，浪费很多时间
2. **"调参"解决不了"认知盲区"**：插入线判定、多格选中失效，不是参数问题，是对问题的理解不够；加上可视化/日志，认知盲区消失，问题立刻可解
3. **有 15 个左右是联动 bug**："修一个引出一个"——修了表格单击闪烁，又引出多格拖拽消失；修了多格拖拽，又影响了行号计算
